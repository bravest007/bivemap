# 3. 요구사항 및 기술 제약 체계

## 4. 비기능 요구사항 (Non-Functional Requirements)

- **극강의 앱 성능 (Performance 최우선)**: 마인드맵 데이터가 1,000~2,000개 이상의 노드로 대량 확장되더라도 메인 렌더 스레드가 전혀 정지하지 않고 60fps 패닝/줌이 유지되어야 함 (`React Flow`의 노드 가상화(Virtualization) 최적화 필수).
- **재사용성 및 모듈 아키텍처 (Recyclability)**: 추후 다른 주제의 맵(주식, 영어 공부, 게임 등)으로 즉시 데이터 JSON만 갈아 끼워 재활용할 수 있게 도메인 데이터와 UI(Map, Jarvis, Ingester)를 철저히 독립 컴포넌트로 분리 설계함.
- **우아한 에러 핸들링 (Error Resilience)**: LLM 환각, API 한도 초과(Rate Limit), 인터넷 단절 등 발생 시, 사용자 시야 밖에서 글로벌 재시도(Retry)를 도입하고 유저에겐 부드러운 토스트 알림을 제공해 앱 화면이 먹통이 되는 프리징 현상을 차단함.
- **모던 디자인 및 반응형 (Aesthetics)**: 바닐라 CSS 사이버네틱 다크 테마(Neon/Glassmorphism) 적용. 데스크탑은 스플릿 뷰(2분할)로 최적화하고, 모바일 기기 접속 시 맵과 챗봇 패널을 상하단 탭으로 토글 처리 가능하도록 반응형 레이아웃 구현.

## 5. 기술 제약사항 (Technical Constraints)

서버나 외부 컴퓨팅에 지속적인 과금이 발생하지 않는 **월 운영 유지비용 $0 (Free-tier 최적화)**을 가장 빡빡한 원칙으로 삼음.

- **프론트엔드 및 호스팅**: `Vercel` 기반 Next.js App Router (Fast SSR 처리 및 Edge 런타임 API 서버리스).
- **LLM 코어 엔진**: `Gemini AI API (Google)`. 무료 티어 내 분당 요청량 제한을 고려하여 텍스트 요약, 번역, 분류 작업, 그리고 JARVIS 응답 기능까지 통합해서 파이프라인 구성.
- **데이터베이스 (지식 저장소 및 RAG 임베딩)**:
  - 기본: `Supabase (PostgreSQL + pgvector)`. 무료 티어로 500MB 한도 내의 지식 데이터와 임베딩 벡터값 저장.
  - **[DB 초과 시 로컬 하이브리드 확장]**: Supabase 무료 용량 한도(500MB) 도달 및 방대한 개인 데이터를 다수 보관해야 할 때를 대비하여, **로컬 컴퓨터 파일 시스템 접근(File System Access API) 또는 브라우저 내장(IndexedDB/Local Storage)**에 직접 지식 파일들과 마인드맵 덩어리를 저장하고 JARVIS가 이를 오프라인으로 참조할 수 있게 하는 브리지(Bridge) 아키텍처를 연동.

## 6. 성공 기준 (Definition of Done - DoD)

이 MVP 단계 개발의 궁극적인 테스트/품질 완전성을 판단하는 최소 3가지 기준:

1. **무결성 주입 컴포넌트 오류 0%**: 무작위 IT 블로그 텍스트 정보 5개를 화면에 연달아 입력했을 때, 백그라운드 파싱 에러나 로직 정지 없이 기존 맵 카테고리에 정확히 의미를 파악하여 5번 연속 연결 생성 렌더링에 성공해야 함.
2. **극한 렌더링 퍼포먼스**: 1,000개가 넘는 노드가 한 화면에 로딩되어 있을 때, 사용자가 이를 드래그하며 빠르게 훑거나 마우스 휠로 대폭 줌 인/줌 아웃을 연속으로 반복해도 브라우저 프레임 저하(버벅거림)가 조금도 체감되지 않아야 함.
3. **지식 기반 RAG 정확도 응답**: 사용자가 방금 업로드해서 맵에 추가해둔 전문 코딩 팁 문서를 가리키며 우측 자비스에게 "저번에 올린 OOO 자료 쉬운 말로 다시 요약해줘"라고 물었을 때, 시스템 환각(거짓 정보) 없이 오로지 내부 로컬/클라우드 저장소 구조만을 참조해 안정되게 10초 내로 답변해야 함.
